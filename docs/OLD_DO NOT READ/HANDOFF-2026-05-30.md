### Bug découvert : pooler cache Supabase
Le Supabase REST API (utilisé par Vercel) et le SQL editor peuvent lire des valeurs différentes pour la même table, probablement à cause du connection pooler (PgBouncer). Exemple : `start_date` montrait `2026-06-11` via REST alors que le SQL editor montrait `2022-11-20`. **Workaround appliqué** : `start_date` est maintenant dérivé des fixtures API dans `sync-fixtures/route.ts` au lieu de la DB.

## 3. Problème en cours : bootstrap retourne `days: 0`

**Symptôme** : `GET /api/competition/bootstrap` retourne :
```json
{ "teams": 30, "days": [], "group_fixtures": [] }
```

Alors que :
- `competition_days` a 23 lignes avec `competition_id = 1`
- `matches` a 43 lignes avec `competition_id = 1`, phase=Groups

**Ce qu'on a testé** :
- Ce n'est pas du cache CDN (cache-buster `?_=timestamp` donne le même résultat)
- Toutes les lignes ont bien `competition_id = 1`
- La query teams fonctionne (30 résultats) mais days et group_fixtures retournent []

**Hypothèse** : La query Supabase `.from('competition_days')` retourne une erreur silencieuse (l'erreur est ignorée avec `const { data: daysRaw } = ...` sans check de l'erreur). Il faut loguer `daysErr`.

**Code à modifier** (`apps/web/app/api/competition/bootstrap/route.ts`, ligne ~92) :
```ts
// AVANT (erreur silencieuse)
const { data: daysRaw } = await adm(admin)
  .from('competition_days')
  ...

// APRÈS (à faire en premier dans la nouvelle session)
const { data: daysRaw, error: daysErr } = await adm(admin)
  .from('competition_days')
  ...
if (daysErr) console.error('[bootstrap] competition_days error:', daysErr);
console.log('[bootstrap] comp.id=', comp.id, 'daysRaw count=', daysRaw?.length ?? 'null');
```

> ⚠️ Ce changement est dans `git stash` — appliquer avec `git stash pop` avant de committer.
